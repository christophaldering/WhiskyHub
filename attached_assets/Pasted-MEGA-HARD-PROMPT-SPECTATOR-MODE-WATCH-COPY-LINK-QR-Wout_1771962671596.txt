MEGA-HARD PROMPT — SPECTATOR MODE (WATCH) + COPY LINK + QR — Wouter/Express/Drizzle

This is a SECURITY-SENSITIVE feature. Implement exactly as specified.
Do NOT expose participant identities. Do NOT allow writes from spectator.
If any acceptance criterion is not met, keep working until it is.

==================================================
GOAL
==================================================
Add a “Spectator / Watch Mode” for tastings:
- View-only access via a separate non-guessable tokenized link.
- Shows LIVE group averages (per dram; optional per dimension) and DRAM RANKING.
- Shows REVEAL when host reveals (and only then).
- Shows NO participant names, NO alias list, NO individual votes, NO notes.
- Provides BOTH: Copy-link button AND QR code in host tools.

This is independent of guest participation modes (standard/ultra).
Spectators must NEVER submit ratings and must not have access to write endpoints.

   - Existing API endpoints for sessions/ratings

Proceed only after the audit summary is printed.

==================================================
PHASE 1 — DATA MODEL (DRIZZLE)
==================================================
Add spectator/watch settings to the session model (use existing session settings table if present; do NOT create parallel tables unless necessary).

Required fields (names may differ, but semantics MUST match):
- viewerEnabled: boolean (default false)
- viewerToken: string (nullable; non-guessable)
- viewerLiveOnly: boolean (default true)
- viewerAllowAverages: boolean (default true)
- viewerAllowRanking: boolean (default true)
OPTIONAL:
- viewerPinHash / viewerPinEnabled (ONLY if you already have PIN infra; otherwise skip for now)

Token requirements:
- viewerToken must be long, random, unguessable (>= 128-bit).
- Generate using crypto-secure random generator on server (Node crypto).
- Never use sessionId as token.

==================================================
PHASE 2 — HOST UI (COPY LINK + QR CODE) — NO NEW SIDEBAR LINKS
==================================================
In the existing Host Dashboard / host tools UI:
Add a “Zuschauer (Watch Mode)” section with:

Controls:
- Toggle: “Zuschauer erlauben” (viewerEnabled)
- Toggle: “Nur während LIVE” (viewerLiveOnly)
- Toggle: “Gruppenmittelwerte anzeigen” (viewerAllowAverages)
- Toggle: “Ranking anzeigen” (viewerAllowRanking)

When viewerEnabled is turned ON:
- Ensure viewerToken exists (create if missing).
- Show:
  1) Copy button: “Zuschauer-Link kopieren”
  2) QR code for the same link
  3) Small info text: “Zuschauer sehen keine Teilnehmer und können nichts eingeben.”

Link format (EXACT semantics):
- /watch/<viewerToken>
(or /session/<id>/watch?token=..., but MUST be tokenized and non-guessable; prefer /watch/<token>)

QR code:
- Generate client-side (small library allowed) OR simple QR component if already available.
- Must encode the exact watch URL.

Important:
- Do NOT add watch mode as a sidebar nav item.
- Only visible to host/admin.

==================================================
PHASE 3 — FRONTEND ROUTE (WOUTER) + WATCH SCREEN
==================================================
Add a Wouter route:
- /watch/:token

WatchScreen requirements:
- No sidebar, no bottom nav, no global header (use a minimal “WatchLayout”).
- Prominent label: “Live Board (Zuschauer)”
- Displays:
  A) Session title (if allowed; otherwise generic “Tasting Session”)
  B) Drams list showing:
     - Dram label (A/B/C or 1/2/3)
     - Group average Gesamt (if viewerAllowAverages)
     - Optional: group averages per dimension (Nosing/Tasting/Abgang/Balance/Gesamt) if enabled
  C) Dram ranking (if viewerAllowRanking):
     - Ranking of DRAMS only (NOT participants).
     - Example: 1) Dram B — 7.8, 2) Dram A — 7.3, etc.
  D) Reveal state:
     - If blind and not revealed: DO NOT show whisky names/images/meta.
     - If revealed and reveal allowed: show whisky name + thumbnail.
- Must auto-refresh (polling) every 2–5 seconds OR use existing realtime mechanism.
- Must be performant: avoid heavy re-renders.

Absolutely forbidden on WatchScreen:
- Participant list
- Participant names/aliases
- Individual ratings
- Notes
- Any input fields that could submit ratings

==================================================
PHASE 4 — BACKEND API (EXPRESS) — READ ONLY
==================================================
Add a read-only endpoint for watch mode, e.g.:
GET /api/watch/:token

This endpoint must:
- Validate token and viewerEnabled=true
- Enforce viewerLiveOnly:
  - If viewerLiveOnly=true, only allow if session is currently live
  - If not live, return 403 with a neutral message
- Return only safe aggregated data:
  - sessionTitle (optional)
  - isBlind, isRevealed (or reveal state for viewers)
  - revealAllowed
  - drams: [ { label, averages (if allowed), revealedWhisky (if revealed+allowed) } ]
  - dramRanking (if allowed)
- MUST NOT return:
  - any participant identifiers
  - any individual votes
  - any notes

Write endpoints protection:
- Ensure spectators cannot call rating submit endpoints successfully.
  Options:
  - Watch token should not authorize any write endpoints.
  - If you have auth middleware, enforce that write endpoints require normal participant auth/token, not viewer token.
  - Add explicit server-side guard: viewer token rejected on write routes.

==================================================
PHASE 5 — BLIND/REVEAL SAFETY (NO LEAKS)
==================================================
In watch mode:
- BEFORE reveal (blind): show only Dram labels and aggregates/ranking by dram.
  No whisky names/images/region/age/abv.
- AFTER reveal: show whisky identity only if reveal is enabled for viewers.

If reveal is disabled by host:
- Never show identity even after session ends.

==================================================
PHASE 6 — HARD ACCEPTANCE CRITERIA (TASK FAILS IF ANY NOT MET)
==================================================
1) Separate watch link exists and is non-guessable tokenized (NOT sessionId).
2) Host can enable/disable watch mode per session.
3) Host UI shows BOTH:
   - Copy link button
   - QR code with exact watch URL
4) Watch screen is view-only:
   - no sidebar/bottom nav/header
   - no inputs
   - no write actions
5) Watch API returns ONLY aggregated data; no participant-level data.
6) Blind safety:
   - before reveal, no whisky identity shown
   - after reveal, identity shown only if allowed
7) Dram ranking is DRAM-based only (no participant ranking).
8) viewerLiveOnly works:
   - when enabled, watch link fails outside live session
9) Rate limiting or caching is applied to watch endpoint (basic protection):
   - at minimum, server-side caching for 2 seconds OR express-rate-limit on that route
10) No console errors; mobile portrait usable.
