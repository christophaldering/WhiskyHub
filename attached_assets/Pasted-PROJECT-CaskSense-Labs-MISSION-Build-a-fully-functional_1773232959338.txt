PROJECT: CaskSense Labs

MISSION
Build a fully functional parallel product experience called **“CaskSense Labs”** inside the existing CaskSense codebase.  
CaskSense Labs must reuse the stable backend, authentication, database models, and services of the current system while introducing a new UX layer and navigation model.

The existing CaskSense application must remain fully operational and unchanged in behavior.

CaskSense Labs is an experimental but real product layer focused on a refined whisky tasting experience built around:

• Host-guided tastings  
• Elegant rating flows  
• Reflection on taste  
• Lightweight community  
• Calm, Apple-inspired design  

NON-NEGOTIABLE SAFETY RULES

1. NEVER break the current CaskSense application.
2. NEVER overwrite existing routes, pages, or core logic.
3. DO NOT refactor stable core systems unless strictly required.
4. All Labs functionality must be additive and isolated.
5. Legacy UX and Labs UX must coexist safely.
6. The current app must still run exactly as before.
7. Labs must reuse backend infrastructure rather than duplicating it.
8. Any schema change must be backward-compatible.
9. Prefer wrapper components over modifying shared components.
10. Validate both systems before completion.

ARCHITECTURE STRATEGY

Implement CaskSense Labs as a **parallel UX layer** inside the same project.

Create a **new route namespace**:

/labs
/labs/join
/labs/host
/labs/solo
/labs/tastings
/labs/tastings/:id
/labs/live/:id
/labs/results/:id
/labs/taste
/labs/circle

Legacy routes must remain untouched.

Labs must reuse the same database, authentication system, user entities, and tasting models.

The Labs implementation should prioritize frontend UX isolation while using the same backend services.

CODEBASE ANALYSIS (FIRST STEP)

Before writing code, analyze the current project and identify:

• routing architecture  
• auth/session management  
• database schema and entities  
• tasting-related models  
• role and permission system  
• shared UI components  
• state management strategy  
• API endpoints  
• server structure  

Adapt the Labs implementation to the existing architecture rather than forcing a new one.

REUSE THESE SYSTEMS

Reuse existing infrastructure whenever stable:

• authentication and session handling  
• user and profile entities  
• role system (guest / participant / host / admin)  
• tasting entities and session structures  
• bottle and distillery entities  
• database connection  
• schema validators and types  
• stable API endpoints  
• shared server services  
• stable UI primitives if helpful

DO NOT duplicate the backend unless absolutely necessary.
