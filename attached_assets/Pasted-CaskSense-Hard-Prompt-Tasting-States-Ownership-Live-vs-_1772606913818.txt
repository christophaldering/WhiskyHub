CaskSense – Hard Prompt: Tasting States, Ownership, Live vs Finished (Spec + Implementation)

Goal:
Implement a clear, minimal state model for tastings that distinguishes:
- live vs not live
- open vs finished
- hosted-by-me vs joined-by-me vs other/public
and derive the necessary UI, API, and data model consequences.
Keep it simple and avoid state explosion.

----------------------------------------
A) DEFINITIONS (single source of truth)
----------------------------------------

1) Tasting lifecycle status (tastings.status):
- created  = session exists, not started
- running  = session active (may be live)
- finished = session closed, read-only

2) Round status (tastings.round_status):
- waiting = no round active yet
- active  = ratings allowed for current whisky
- closed  = round ended; ratings locked for that whisky

3) Live definition:
A tasting is considered "LIVE" iff:
tastings.status = running AND tastings.round_status = active
(Optionally: host is currently controlling rounds)

4) Ownership definitions:
- Hosted by me: tastings.host_user_id = current_user.id
- Joined by me: exists participants row with participants.user_id = current_user.id for that tasting
- Other/public: neither hosted nor joined, but tastings.is_public = true (optional)

5) Past vs future:
Do NOT create separate states for past/future.
Use timestamps:
- tastings.scheduled_at (optional)
- tastings.finished_at (recommended)
Past = finished_at is set OR status=finished
Future = scheduled_at in the future (optional)

----------------------------------------
B) DATA MODEL CHANGES (Postgres/Drizzle)
----------------------------------------

1) Add/ensure fields on tastings:
- status enum('created','running','finished') default 'created'
- round_status enum('waiting','active','closed') default 'waiting'
- current_whisky_id (nullable FK to tasting_whiskies)
- results_visible boolean default false
- reveal_done boolean default false
- scheduled_at timestamp nullable
- finished_at timestamp nullable

2) Permissions-relevant fields:
- host_user_id nullable FK to users

3) Do NOT add more status flags beyond this in V1.

----------------------------------------
C) BUSINESS RULES (must be enforced server-side)
----------------------------------------

1) Start tasting:
Host action sets:
status = running
round_status = waiting (until first round starts)

2) Start round (host-only):
round_status = active
current_whisky_id must be set
results_visible default false
reveal_done default false (especially in blind mode)

3) End round (host-only):
round_status = closed
Lock ratings for current whisky (ratings.locked=true) OR enforce via server rule.

4) Finish tasting (host-only):
status = finished
finished_at = now()
round_status can remain closed or set to waiting; but ratings must be read-only.
No new ratings submissions allowed when status=finished (V1).

5) Rating submission rules:
Allow POST /ratings only if:
status = running AND round_status = active
Otherwise return 409 with reason.

6) Reveal rules (host-only):
If blind_mode=true, participants see only display_label until reveal_done=true.
After reveal_done=true, participants can see whisky_name_snapshot (and meta).

7) Results visibility:
Participants can only see aggregates (avg, count) if results_visible=true OR status=finished.

----------------------------------------
D) UI CONSEQUENCES (Front-end structure)
----------------------------------------

1) My Taste → Tastings screen must support filters/tabs:

- Hosted (host_user_id = me)
- Joined (participant user_id = me)

Within each tab add filters:
- Open (status in created,running)
- Live now (status=running AND round_status=active)
- Finished (status=finished)

2) Host Dashboard
Only show sessions that are hosted-by-me.
Priority sections:
- Active/Live at top
- Created (not started) below
- Finished in history

3) Join Flow (participants)
After joining by session code:
- Show current dram label (A/B/C) if blind_mode and not revealed
- Enable rating UI only if status=running AND round_status=active
- If round closed: read-only view + waiting message

4) No login friction on landing.
Join remains guest-friendly.

----------------------------------------
E) API CONSEQUENCES (endpoints + semantics)
----------------------------------------

1) Tasting queries:
- GET /api/tastings/my/hosted?status=open|live|finished
- GET /api/tastings/my/joined?status=open|live|finished
(Implement as server filters; frontend can also filter but server is preferred.)

2) State endpoints (host-only write):
- GET  /api/tastings/:id/state
- POST /api/tastings/:id/state
Validate host permissions.

3) Join endpoint:
- POST /api/tastings/:id/join
Creates participant row (guest or user)
Returns participant_id and current state snapshot.

4) Ratings:
- POST /api/ratings/submit (enforce state rules)
- GET  /api/tastings/:id/results (only if results_visible=true or status=finished)

----------------------------------------
F) IMPLEMENTATION STRATEGY (V1)
----------------------------------------

1) Keep state machine minimal: created → running → finished
2) Live is derived, not stored separately.
3) Past/future derived from scheduled_at/finished_at, not separate states.
4) V1: Disallow late ratings after finished to keep logic stable.
5) Use polling (React Query refetchInterval) for LIVE state updates.

Deliverables:
- Drizzle schema updates + migrations
- Server-side rule enforcement
- My Taste tabs/filters for hosted/joined/open/live/finished
- Host dashboard prioritization of live sessions
- Participant UI gating based on state
