## TEIL 2: SKALEN-SYSTEM (FUNDAMENTALE GRUNDLAGE)

**Diese Logik MUSS zuerst implementiert werden, da alles andere darauf aufbaut.**

### 2.1 Skalen-Hierarchie (unveränderlich)
```
Host-definierte Skala > Nutzer-Profilpräferenz > Default (100)
```

### 2.2 Unterstützte Skalen
| Skala | Max | Schrittweite Fein | Schrittweite Grob | Dial-Schritte |
|-------|-----|-------------------|-------------------|---------------|
| 100   | 100 | 1                 | 5                 | fließend      |
| 20    | 20  | 1                 | 2                 | 20 Positionen |
| 10    | 10  | 1                 | 1                 | 10 Positionen |
| 5     | 5   | 1                 | 1                 | 5 Positionen  |

### 2.3 Nutzer-Profilpräferenz
- In `client/src/labs/pages/LabsProfile.tsx` (oder äquivalent): Dropdown/Segmented Control zur Auswahl der bevorzugten Skala
- Gespeichert via `PATCH /api/participants/:id` auf `preferredRatingScale`
- Feld `preferredRatingScale` muss in `participants`-Tabelle vorhanden sein (Migration prüfen!)

### 2.4 Skalen-Hook
Erstelle `client/src/labs/hooks/useRatingScale.ts`:
```typescript
// Gibt die aktive Skala zurück:
// - tastingScale (wenn in aktivem Tasting) überschreibt alles
// - sonst participantScale (aus Profil)
// - sonst default: 100
export function useRatingScale(tastingId?: string): {
  max: number;
  step: number;
  bigStep: number;
  label: string;
  normalize: (val: number) => number; // immer auf 0-100 für DB
  denormalize: (normalized: number) => number; // zurück zur Anzeige
}
```

### 2.5 Normalisierung
- **Alle Scores werden in der DB IMMER als 0-100 gespeichert** (bestehende Logik beibehalten)
- Anzeige und Eingabe erfolgen in der nativen Skala
- Konvertierung passiert im Hook, nicht in der UI

## TEIL 3: RATING DIAL KOMPONENTE

### 3.1 `RatingDial.tsx` – Spezifikation

```typescript
interface RatingDialProps {
  value: number;           // nativer Skalenwert (z.B. 0-100 oder 0-10)
  onChange: (v: number) => void;
  scale: RatingScale;      // aus useRatingScale
  size?: 'sm' | 'md' | 'lg'; // sm=140px, md=190px (default), lg=220px
  disabled?: boolean;
}
```

**Pflichtverhalten:**
- Canvas-basierter Arc-Dial (270° Bogen, startet unten links)
- Goldener (#c8861a) ausgefüllter Bogen + goldener Thumb-Punkt
- Grauer Track als Hintergrund
- Drag-Interaktion: Maus UND Touch (mit `touch-action: none`)
- Zahl in der Mitte: native Skala, groß (48px, font-weight 500)
- Subtext: `/100` bzw. `/20` etc.
- Bump-Animation bei Wertänderung (scale 1.1, 120ms)
- Bei diskreten Skalen (≤20): Snap auf nächste gültige Position
- ±1 und ±BigStep Buttons UNTERHALB des Dials (nicht daneben)
- Button-Layout: `[−Big] [−1] [Feinabstimmung] [+1] [+Big]` in Pill-Form
- `cursor: grab`, `user-select: none`

**VERBOTEN:**
- Kein HTML-Slider (`<input type="range">`) als primäre Eingabe
- Keine Darstellung auf einer Linie mit dem Dial

## TEIL 4: SOLO-BEWERTUNGSFLOW

### 4.1 Entry Point: Nach Whisky-Anlage

In `LabsTastingsSolo.tsx`:
- Nach erfolgreichem Speichern eines neuen Whiskys (`POST /api/journal`) → **sofortiger, automatischer** Wechsel in den Bewertungsflow (kein Extra-Tap)
- State: `soloView` erweitern um `'rating'`
- Übergabe: `{ whiskeyId, whiskyName, whiskyMeta }` an `RatingFlow`

### 4.2 Unterbrechungs-Banner (`ResumeRatingBanner.tsx`)

Wenn Nutzer zurückkommt und eine unvollständige Bewertung existiert:
- **Kein Modal, kein Popup**
- Dezentes Banner AM TOP der Whisky-Detailansicht
- Inhalt: „[WhiskyName] bewerten – du warst bei [Nose/Taste/Finish]"
- Zwei Aktionen: „Weiter bewerten" (→ RatingFlow ab letztem Step) | „×" (schließen, wegwischbar)
- Prüfung: Bewertung unvollständig = mind. eine Dimension ist null/undefined
- Banner verschwindet nach vollständigem Speichern

### 4.3 Sequenzieller Flow (`RatingFlow.tsx`)

**Flow-Schritte:**
```
Step 0: Nose
Step 1: Taste  
Step 2: Finish
Step 3: Summary + Flavours + Speichern
```

**Navigation (PFLICHT):**
- UNTEN: `[← Zurück]` (links, sekundär) + `[Weiter zu Taste →]` (rechts, gold, primär)
- Zurück-Button auf Step 0: disabled (nicht versteckt)
- Dots oben: NUR informativ, NICHT klickbar
- Auf Summary (Step 3): Recap-Pills für Nose/Taste/Finish sind klickbar → springen zu Step, `btn-next` wechselt zu „Zurück zum Ergebnis ↩"
- `returnToSummary`-Flag: wenn gesetzt, geht Weiter direkt zur Summary statt zum nächsten Schritt

**Pro Step (Nose/Taste/Finish):**
- Dimension-Label (UPPERCASE, letter-spacing, muted)
- Hint-Text (was tun in diesem Schritt)
- `RatingDial` (Größe md)
- Adj-Buttons darunter

**Summary (Step 3):**
- Recap-Row: Nose | Taste | Finish als klickbare Pills (tippen zum Korrigieren)
- Hint: „Tippen zum Korrigieren" (tiny, muted)
- `OverallCircle` (animiert einlaufend, 80ms delay nach Render)
- Überschreiben-Button → Zahlen-Input erscheint inline (kein Modal)
- Zurücksetzen-Button → setzt auf Mittelwert zurück
- `FlavourPicker` (kategorisiert, 3 Kategorien sichtbar, Rest aufklappbar)
- Notizen-Textarea (optional, 2 Zeilen, expandierbar)
- Bottom-Nav: `[← Zurück]` + `[Speichern ✓]` (gold)

**Nach Speichern:**
- Keine neue Seite
- Overlay INNERHALB der Komponente erscheint (nicht fixed/modal)
- Inhalt: Checkmark-Animation, Bestätigung, dein Score, Optionen
- Optionen: „Korrigieren" (→ Step 3) | „Neuen Dram anlegen" | „Zur Übersicht"

### 4.4 `OverallCircle.tsx`

```typescript
interface OverallCircleProps {
  nose: number | null;
  taste: number | null;
  finish: number | null;
  override?: number | null;
  scale: RatingScale;
  animate?: boolean; // default true
  size?: number;     // px, default 80
}
```

- Berechnung: `Math.round((nose + taste + finish) / 3)` wenn kein Override
- Animation: Bogen läuft von 0 auf Zielwert in ~600ms (requestAnimationFrame)
- Override: Nutzer tippt „Überschreiben" → Zahl editierbar → Kreis zeigt manuellen Wert
- „Zurücksetzen" → zurück zum berechneten Mittelwert

### 5.1 `FlavourPicker.tsx`

**Pflichtverhalten:**
- Kategorisierte Darstellung (nicht flach)
- Standard: **3 Kategorien sichtbar**, passend zum Whisky-Typ (wenn bekannt)

- Ganz unten im aufgeklappten Bereich: Experten-Tools als winzige Pills (Aromerad, Compass, Regionen) – dezent, kaum sichtbar
- Chip-State: inaktiv (grau) / aktiv (gold)
- Kategorie-Zähler: zeigt Anzahl aktiver Chips pro Kategorie (z.B. „3 ✓")
- Chips sind Tags: Auswahl wird in `selectedFlavours: string[]` gespeichert

**Kategorien (diese 6 primär, 3 initial sichtbar):**
```
Sichtbar: Fruchtig | Süß & Würzig | Holz & Leder
Versteckt: Rauch & Torf | Nussig & Malzig | Maritim & Floral
```

**Kontext-Anpassung (wenn Whisky-Metadaten bekannt):**
- Islay/Torf/Rauch → „Rauch & Torf" nach oben schieben

- Maritim/Islands → „Maritim & Floral" nach oben
- **Im Blind-Modus: KEINE Kontext-Anpassung** (verhindert Metadata-Leaks)

## TEIL 6: TASTING-TEILNEHMER-FLOW

### 6.1 Kontext-Bar (oben, immer sichtbar)

```
[Dram X von N] · [Tasting-Name] · [Skala-Badge] · [Blind-Badge falls blind]
```

- Skala-Badge: „Skala 1–100" etc. in Gold-Pill
- Einmalig sichtbar beim ersten Dram, dann persistent

### 6.2 Dram-Karussell (`DramCarousel.tsx`)

- Horizontal scrollbar, `overflow-x: auto`, kein Scrollbar sichtbar
- Pro Dram: kleiner Card-Chip mit Nummer, Kurzname, Score (wenn bewertet)
- Status-Klassen: `done` (gold-dim) | `active` (gold) | `locked` (opacity 0.35)
- Blind-Modus: gesperrte Dramen zeigen „Sample X", keinen echten Namen
- Badge bei offenen Bewertungen: kleines „!" oben rechts am Chip
- Tap auf Done-Chip: öffnet Zusammenfassung für diesen Dram (read-only oder editierbar je nach Status)

### 6.3 RatingFlow im Tasting-Kontext

**Identisch zu Solo**, mit diesen Unterschieden:
- `DramCarousel` oben statt Whisky-Header
- Kontext-Bar ganz oben
- Nach Speichern: KEIN „Neuen Dram anlegen" – stattdessen „Warte auf Host für Dram X+1"
- Gruppen-Score auf der After-Save-Ansicht: **nur sichtbar nach vollständigem Reveal** des aktuellen Drams
- Vergleich mit eigenen vorherigen Dramen immer sichtbar
- Auto-Save: alle 30s, auch wenn Flow nicht abgeschlossen (verhindert Datenverlust)

### 6.4 Dram-Wechsel (SSE-getriggert)

Wenn `status_changed` oder Host `guidedWhiskyIndex` ändert:
1. Wenn aktueller Dram VOLLSTÄNDIG bewertet: sanfte Transition direkt zu neuem Dram
2. Wenn aktueller Dram UNVOLLSTÄNDIG: `ResumeOrSkipBanner` erscheint
   - „Jetzt speichern" (speichert was da ist) | „Überspringen"
   - Banner zeigt was bereits eingetragen ist (z.B. „Nose: 74, Taste: –, Finish: –")
   - Bisherige Eingaben gehen NICHT verloren
3. Nach Banner-Entscheidung: Übergang zu neuem Dram mit `popIn`-Animation

## TEIL 7: HOST-COCKPIT ERWEITERUNGEN

### 7.1 Reveal-Flow im Cockpit

**Linke Spalte – Reveal-Stepper:**

```
[Preset-Auswahl: Classic | Name | Foto | Einzeln]
[Stage-Pills: Pill1 | Pill2 | Pill3] (done/active/pending)
[▶ Reveal starten / ▶ Weiter: [NextStageName] / Alle Infos gezeigt ✓]
[Sichtbarkeits-Hint:]
  Host sieht: [Alles (Whiskyname)]
  Teilnehmer: [Was aktuell sichtbar ist]
```

**Preset-Logik:**
- Classic: Name → Details → Foto
- Name zuerst: Name → Foto → Details  
- Foto zuerst: Foto → Name → Details
- Einzeln: Name → Distillerie → Alter → ABV → Foto

**Reveal-Button:**
- Löst SSE-Event `reveal_triggered` aus (Backend: vorhandene Logik nutzen)
- Flash-Effekt: weißes Overlay, opacity 0.65, 180ms (alle verbundenen Clients)
- Nach letztem Schritt: Gruppen-Scores werden in mittlerer Spalte freigeschaltet

**Rechte Spalte – Live Guest Preview:**
- Zeigt LIVE was Teilnehmer aktuell sehen (basierend auf revealStep)
- Aktualisiert sich synchron mit jedem Reveal-Klick (kein Re-Fetch nötig, lokaler State)
- Blind: graues `?` Placeholder
- Nach Name-Reveal: Name animiert eingeblendet (`popIn`)
- Nach Details: Detail-Grid erscheint (`slideUp`)
- Nach Foto: Emoji/Foto groß (`fadeScale`)
- Gruppen-Scores: erscheinen erst NACH vollständigem Reveal

### 7.2 Nächster-Dram-Flow

**Bestätigungs-Dialog (KEIN Modal – inline in linker Spalte):**
```
Weiter zu Dram [N+1]?
[Whisky-Name] wird als nächster Dram geöffnet.

⚠ [Name] hat Dram [N] noch nicht bewertet.  ← nur wenn jemand unvollständig

[Abbrechen]  [Ja, weiter zu Dram N+1 →]
```

**Nach Bestätigung:**
1. Backend: `PATCH /api/tastings/:id/guided-advance` (vorhandener Endpoint)
2. SSE-Event an alle Clients
3. Lineup in linker Spalte: alter Dram → `done`, neuer Dram → `current`
4. Mittlere Spalte: Fortschrittsanzeige für neuen Dram, Score-Balken leer
5. Rechte Spalte: Guest-Preview zeigt neuen Dram blind

### 7.3 Mobile Host-Tabs

Auf Mobile (< 768px) – drei Tabs innerhalb der Host-Ansicht:
- **Übersicht**: Lineup + Teilnehmer-Status (Completion)
- **Meine Bewertung**: Host-eigener RatingFlow (gleiche Komponente wie Teilnehmer)
- **Gruppe**: Echtzeit-Durchschnitte + Varianz + Konsens-Indikatoren

- Connoisseur Report wird nach jedem abgeschlossenen Tasting angeboten

**Generierungs-Moment:**
- Kurze Anzeige (ca. 2s): animierter Puls-Dot + Text „KI analysiert deine Verkostung…"
- „Sofort anzeigen"-Link überspringt Warte-Animation
- KEIN Blocking-Spinner, KEIN Modal

**Cover-Bereich:**
- Eyebrow: „Connoisseur Report"
- Titel: KI-generierter, prägnanter Charaktersatz (kein generisches „Dein Score war X")
- Subtext: Kontext (Tasting-Name, Anzahl Dramen, Datum, Ø Score)
- Meta-Pills: Tasting-Name | Anzahl Dramen | Datum | Ø Score
- DE/EN-Toggle oben rechts (Segmented, Gold bei aktiver Sprache)
- Sprach-Wechsel aktualisiert alle KI-Texte sofort (kein Re-Fetch, beide Versionen werden beim Generate erstellt)

**Stat-Row (4 Karten):**
Ø Score | +/- vs. Gruppe | Höchster Score | Niedrigster Score

**Radar-Chart:**
- Canvas-basiert (kein externe Library-Abhängigkeit)
- 5 Dimensionen: Nose, Taste, Finish, Balance*, Overall (*falls vorhanden)
- Zwei Polygone: Du (gold) + Gruppe (grau, transparent)
- Legende rechts mit Bar-Balken

**4 Tabs:**

| Tab | Inhalt |
|-----|--------|
| Bericht | 3 KI-Abschnitte + Zitat-Block + Teilen-Karte + Aktions-Buttons |
| Whiskys | Ranking mit Score, Δ Gruppe, Aromen-Chips pro Flasche |
| Aromen | Flavour-Cloud (Chip-Größe = Häufigkeit) für dieses Tasting + Gesamtprofil |
| Verlauf | Balken-Chart: Ø Score + Δ Gruppe über letzte 5 Tastings |

**Teilen-Karte:**
- Vorschau-Zitat (KI-generiert, 1-2 Sätze)
- Aktionen: PDF | Kopieren | Teilen
- PDF-Export via jsPDF (bestehende Logik nutzen/erweitern)

**Aktionen unten:**
- „Frühere Berichte" → Liste aller generierten Reports
- „Empfehlungen" → navigiert zu `LabsRecommendations`

### 8.3 Backend-Anforderungen

- `POST /api/participants/:id/connoisseur-report`: akzeptiert `tastingId` als optionalen Parameter (zusätzlich zu allgemeinem Report)
- Wenn `tastingId` gesetzt: Report fokussiert auf dieses Tasting + historischen Kontext
- Beide Sprachen (DE/EN) werden in einem API-Call generiert und gespeichert
- `dataSnapshot` enthält: alle Ratings des Tastings, Gruppen-Durchschnitte, Aromen, historische Vergleiche

## TEIL 9: ADMIN – FLAVOUR-LOGIK-EDITOR

### 9.1 Anforderung

Admin kann die Flavour-Kategorien und Deskriptoren bearbeiten. Zugang: Admin-Panel → Tab „Aromen".

### 9.2 Datenstruktur

```typescript
interface FlavourCategory {
  id: string;
  nameDE: string;
  nameEN: string;
  sortOrder: number;
  subgroups?: FlavourSubgroup[];
  descriptors?: FlavourDescriptor[]; // wenn keine Subgroups
}

interface FlavourSubgroup {
  id: string;
  nameDE: string;
  nameEN: string;
  sortOrder: number;
  descriptors: FlavourDescriptor[];
}

interface FlavourDescriptor {
  id: string;
  nameDE: string;
  nameEN: string;
  sortOrder: number;
}
```

### 9.3 Persistenz

- Speicherung in DB: neue Tabelle `flavour_categories` + `flavour_descriptors`
- Fallback: wenn DB leer → lade aus `flavor-data.ts` (bestehende Hardcode-Daten)
- API: `GET/POST/PATCH/DELETE /api/admin/flavour-categories`
- Migration: Seed-Script liest bestehende `flavor-data.ts` und befüllt DB

### 9.4 Admin-UI

In `LabsAdmin.tsx` neuer Tab „Aromen":
- Liste aller Kategorien (drag-reorder für sortOrder)
