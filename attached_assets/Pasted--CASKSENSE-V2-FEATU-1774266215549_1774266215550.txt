CaskSense V2 läuft vollständig. Die GPT-4o Flaschenerkennung ist bereits
in der App integriert — sie scannt ein Foto und gibt strukturierte
Whisky-Daten zurück (Name, Destillerie, Region, Age, ABV, Cask).

Das Problem: Wenn die KI falsch liegt, hat der Nutzer keine Möglichkeit
zu korrigieren. Korrekturen gehen verloren. Die Erkennungsqualität
verbessert sich nicht.

Dieses Feature ergänzt direkt nach der GPT-4o-Antwort einen
Feedback-Schritt: Nutzer bestätigt, korrigiert oder lehnt ab.
Korrekturen werden gespeichert und können die KI-Qualität verbessern.

AUFGABE — SCHRITT 1: SERVER FEEDBACK-ENDPUNKT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

In server/routes.ts — NUR diesen einen neuen Endpunkt ergänzen:

POST /api/bottle-recognition/feedback
  Header: x-participant-id (optional, für Logging)
  Body:
    {
      original:      BottleRecognitionResult,
      corrected?:    BottleRecognitionResult,   // nur bei action='corrected'
      action:        'confirmed' | 'corrected' | 'rejected',
      participantId: string | null,
      timestamp:     string (ISO)
    }

  Verarbeitung:
    - Body in JSON parsen und in eine In-Memory Liste pushen
      (server-seitige Variable: const feedbackLog: any[] = [])
    - action='corrected': console.log mit original vs corrected für spätere Auswertung
    - 200 { ok: true } zurückgeben

  WICHTIG: Dieser Endpunkt ist bewusst minimal. Kein DB-Write, kein Schema.
  Die Logs können später manuell ausgewertet oder in eine DB migriert werden.
  Der Endpunkt MUSS funktionieren bevor die Komponente eingebunden wird —
  aber die Komponente fällt auch ohne ihn nicht aus (localStorage-Fallback).

──────────────────────────────────────────────────────────────────────────────
FEEDBACK ABSENDEN (intern)
──────────────────────────────────────────────────────────────────────────────

handleConfirm():
  1. saving = true
  2. POST /api/bottle-recognition/feedback mit { original, action: 'confirmed', ... }
     Bei Netzwerkfehler: localStorage 'cs-recognition-feedback' als JSON-Array ergänzen
  3. saving = false
  4. showFlash = true → nach 300ms: onConfirm(recognition)

handleSaveEdit():
  1. saving = true
  2. POST mit { original, corrected: edited, action: 'corrected', ... }
     Bei Fehler: localStorage-Fallback
  3. saving = false
  4. showFlash = true → nach 300ms: onConfirm(edited)

handleReject():
  1. POST mit { original, action: 'rejected', ... }
     Bei Fehler: localStorage-Fallback
  2. mode = 'rejected'

Header x-participant-id immer mitsenden wenn participantId vorhanden.

Die bestehende Komponente die GPT-4o für Flaschenerkennung aufruft
(suche nach: fetch('/api/recognize') oder fetch('/api/bottle') oder
ähnlichen Aufrufen mit einem Foto/Bild-Upload in labs-v2):

NUR ergänzen — keine bestehende Logik verändern:

  // Import ergänzen:
  import { BottleRecognitionFeedback, BottleRecognitionResult }
    from '../components/BottleRecognitionFeedback'

  // State ergänzen (neben bestehenden States):
  const [recognition, setRecognition] = useState<BottleRecognitionResult | null>(null)

  // Nach dem GPT-4o-Aufruf — in den bestehenden success-Handler ergänzen:
  // (GPT-Antwort in BottleRecognitionResult mappen)
  setRecognition({
    name:       gptResult.name       ?? '',
    distillery: gptResult.distillery ?? '',
    region:     gptResult.region     ?? '',
    age:        typeof gptResult.age === 'number' ? gptResult.age : null,
    abv:        typeof gptResult.abv === 'number' ? gptResult.abv : null,
    caskType:   gptResult.caskType   ?? gptResult.cask ?? '',
    confidence: typeof gptResult.confidence === 'number' ? gptResult.confidence : 0.5,
    rawLabel:   gptResult.rawLabel   ?? gptResult.label ?? '',
  })

  // Im JSX — NACH dem bestehenden Scan-UI, als bedingtes Element:
  {recognition && (
    <BottleRecognitionFeedback
      th={th} t={t}
      recognition={recognition}
      participantId={participantId}
      onConfirm={(data) => {
        // Hier die bestehende Logik aufrufen die einen Whisky anlegt/auswählt
        // (z.B. applyRecognizedBottle(data) oder setWhiskyData(data))
        // Die genaue Funktion hängt vom bestehenden Code ab — den passenden
        // Aufruf aus dem vorhandenen success-Handler übernehmen.
        setRecognition(null)
      }}
      onDismiss={() => setRecognition(null)}
    />
  )}

WICHTIG: Falls die GPT-4o-Antwort andere Feldnamen hat als hier angenommen,
die Mapping-Logik (setRecognition-Aufruf) entsprechend anpassen.
Niemals die GPT-4o-Aufruf-Logik selbst verändern.
