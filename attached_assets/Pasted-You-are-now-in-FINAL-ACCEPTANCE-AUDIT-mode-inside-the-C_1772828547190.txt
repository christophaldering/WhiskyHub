PHASE 3 — LEAK PREVENTION CHECKLIST
======================================================================

You must explicitly check for the following common leakage patterns and fix any issue found:

1) Hidden UI but open API
- UI hides archive, but endpoint still returns restricted payload

2) Partial metadata leakage
- archive list hidden, but counts, titles, community names, tasting numbers, winners, or slugs leak elsewhere

3) Prefetch/query leakage
- client prefetches restricted data before authorization is confirmed

4) Debug/admin leakage
- internal routes accessible without admin/internal protection

5) Overbroad analytics
- “public aggregated” endpoint accidentally includes identifiable or community-specific detail

6) Mixed authorization branches
- one endpoint checks auth correctly, another related one does not

7) Server-side filter mistakes
- endpoint returns all rows then filters only in frontend

8) Direct ID enumeration
- guessing tasting IDs/community IDs reveals restricted content

9) Type-level leakage
- response DTOs include restricted fields that the UI currently ignores, but are still returned over the wire

10) Caching leakage
- React Query or server cache allows restricted data to persist across auth changes or user contexts

For every issue found:
- fix it
- re-test it
- document it in the final report
