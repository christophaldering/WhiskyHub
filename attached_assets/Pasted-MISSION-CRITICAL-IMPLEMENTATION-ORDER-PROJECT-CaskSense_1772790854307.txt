==================================================
1. PRODUCT GOAL
==================================================

When a host has started a live tasting, the host must be able to open a dedicated Hosting Dashboard.

This Hosting Dashboard is the operational control center of the running tasting.

It must allow the host to:
- understand the current tasting status at a glance
- control the tasting flow in real time
- see lineup, participant state, progress, and session status
- handle blind tasting workflows with sequential reveal of prepared information
- moderate the tasting elegantly and safely
- separately enter or access the host’s own rating
- see a live preview of what participants currently see
- cleanly close the tasting
- trigger or open results / evaluation flow
- retain a strong feeling of calm, control, clarity, and premium product quality

This must feel like a high-end live control center.
Not like a crude admin page.
Not like a developer tool.
Not like an overloaded enterprise backend.

4. MANDATORY FEATURE SCOPE
==================================================

The Hosting Dashboard must contain at minimum the following capabilities and modules.

--------------------------------------
4A. SESSION HEADER
--------------------------------------
Must show at minimum:
- tasting title
- live session status
- host role/status
- mode (blind / non-blind, where applicable)
- participant count
- current flight / sample position
- overall progress
- session/join reference if relevant and already part of the system
- clear primary actions such as start / continue / end where contextually relevant

The header must immediately create orientation and confidence.

--------------------------------------
4B. LINEUP / FLOW OVERVIEW
--------------------------------------
Must provide:
- full lineup or current structured flow overview
- clear highlighting of current sample / current flight
- separation of previous / current / upcoming
- visible state per sample/flight where meaningful
- for blind tastings: distinction between host-internal knowledge and participant-visible stage

The lineup must feel curated and intentional, not like a rough list.

--------------------------------------
4C. PARTICIPANT STATUS PANEL
--------------------------------------
Must show:
- participant list
- meaningful session state per participant where technically possible
- such as joined / pending / active / rating / completed / inactive / disconnected
- who is done
- who is still open
- aggregated or per-participant progress where sensible

This must be readable, calm, compact, and useful during moderation.

--------------------------------------
4D. LIVE CONTROL PANEL
--------------------------------------
The host must be able to control the tasting from the dashboard, including at minimum:
- start tasting
- continue tasting
- open rating phase
- close rating phase
- activate next sample / next flight
- trigger reveal step(s)
- move toward closing
- trigger results/evaluation availability
- end session

Controls must be understandable and orderly.
No confusing clusters of equal-weight buttons.

--------------------------------------
4E. BLIND TASTING CONTROL
--------------------------------------
This is a critical requirement.

The host must be able to reveal prepared information sequentially.

The system must support a staged reveal model, for example:
- hidden
- code only
- partial reveal
- more details
- full reveal
- results open

Exact naming can differ, but the logic must be explicit, visible, and testable.

The dashboard must show:
- what the host knows internally
- what participants currently see
- what reveal step is active
- what reveal actions are available next

Reveal actions must be clear and safe.
If a step is effectively irreversible, the interface must communicate that appropriately.

No accidental early exposure of blind information.

--------------------------------------
4F. HOST RATING PANEL
--------------------------------------
The host must have a separate space to enter/access their own rating.

This must NOT be mixed into the main session-control area.

The host has a dual role:
- moderator / controller
- participant / rater

The UI must separate these roles cleanly.

Acceptable forms:
- dedicated side panel
- dedicated card region
- drawer/panel with permanent or easy access
- dedicated subview within the dashboard area

But the separation must be obvious and functional.

--------------------------------------
4G. PARTICIPANT VIEW PREVIEW
--------------------------------------
A separate panel/window/preview area is mandatory.

The host must be able to see what participants currently see.

This is essential.

It must reflect at minimum:
- current participant step/state
- visible information
- hidden/not-yet-revealed information
- relevant CTA or rating state from participant perspective

This preview must reduce uncertainty and support confident moderation.

--------------------------------------
4H. CLOSING / RESULTS TRANSITION
--------------------------------------
The dashboard must support the clean ending of the tasting:
- close final rating phase
- complete reveal if appropriate
- mark tasting as ended
- trigger or unlock results/evaluation
- display final session state

The transition from live tasting to completion/results must feel intentional and productized.
